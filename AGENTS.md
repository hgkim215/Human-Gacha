# Human-Gacha 에이전트 가이드

## 프로젝트 목표

Human-Gacha는 한국어 제품 콘셉트인 **인간 가챠**를 Flutter로 구현하는
demo-first MVP입니다. 사용자는 매일 인간 등급을 뽑고, 오늘의 미션을 받은 뒤,
가벼운 친구방 피드에 결과를 공유합니다.

제품 판단의 기준 문서는 `docs/product/인간_가챠_PRD_v2_demo_first.md`입니다.
P1/P2 운영 기능보다 P0 데모 가치와 발표 안정성을 먼저 만듭니다.

## 제품 핵심

- 한 줄 정의: 사용자의 현재 상태를 몇 번의 선택으로 입력하면 AI가 인간 등급과
  오늘의 미션을 뽑아주는 B급 자기관리 가챠 서비스입니다.
- 핵심 은유: 목표를 직접 쓰는 앱이 아니라 **오늘의 목표를 뽑는 앱**입니다.
- 핵심 루프: 앱 실행 → 오늘의 픽업 확인 → 상태 선택 → 인간 가챠 뽑기 →
  AI 인간 카드 → 미션 수행 → 뱃지/실패 칭호 획득 → 친구방 공유입니다.
- 핵심 차별점: 높은 등급은 최종 보상이 아니라 `후보`이고, 미션 성공 시 뱃지가
  확정됩니다. 실패해도 칭호와 자아 파편이 남아 콘텐츠가 됩니다.
- 제품 톤: 진지한 갓생 관리가 아니라 "오늘 하루 덜 망한 인간을 뽑는" 가벼운
  B급 감성입니다. 웃기되 사용자를 심하게 비하하지 않습니다.

## 반드시 지킬 PRD 기준

- 첫 10초 안에 인간 가챠의 핵심 화면이 보여야 합니다.
- 첫 30초 안에 오늘의 픽업 인간, 확률 버프, 오늘 상태 선택, 뽑기 버튼,
  뽑기 연출 시작이 보여야 합니다.
- 90초 데모 안에 컨셉, 재미, AI 개인화, 미션 보상, 친구방 공유 가능성이 모두
  전달되어야 합니다.
- 핵심 화면은 캡처했을 때 공유 가능한 장면이어야 합니다.
- AI 호출, 네트워크, 인증, 원격 저장이 실패해도 데모 루프는 끝까지 진행되어야
  합니다.

## P0 데모 필수 범위

P0 작업은 아래 기능을 먼저 닫는 방향으로 쪼갭니다.

- Demo User 진입: 앱 실행 후 로그인 없이 바로 홈으로 갈 수 있어야 합니다.
- 온보딩 선택: 4개 이하 질문, 버튼 선택만 사용합니다.
- 오늘 상태 선택: 1~3개 선택 후 바로 뽑기로 이어집니다.
- 오늘의 픽업 인간과 확률 버프: 실제 확률보다 기대감 전달이 우선입니다.
- 인간 가챠 뽑기: 버튼 1개로 연출이 시작되어야 합니다.
- 뽑기 연출: 최소 3단계 이상으로 등급 공개 전 기대감을 만듭니다.
- AI 인간 카드: 사용자 상태와 등급을 반영하되 5초 내 fallback이 가능해야 합니다.
- 미션 완료/실패 처리: 둘 다 보상 화면으로 이어져야 합니다.
- 친구방/전체방 더미 피드: 실제 소셜이 아니라 바이럴 가능성을 보여주는
  발표용 무대입니다.
- Demo Mode/Seed: SR/SSR/실패 시나리오를 안정적으로 재현할 수 있어야 합니다.

## 90초 데모 기준 화면 순서

기능과 화면을 설계할 때 아래 발표 흐름을 기본 경로로 둡니다.

1. Home: 오늘의 픽업 인간, 확률 버프, 뽑기 버튼
2. 상태 선택: `잠 부족`, `카페인 필요`, `코테 준비 중`
3. Draw Animation: 인간성 스캔, 등급 실루엣, SR 공개
4. Result Card: `SR 계획적인 취준 인간 후보`, 미션, 실패 칭호
5. Mission Complete: 완료 버튼, 뱃지 획득
6. Friend Room: 자동 공유 포스트, 리액션
7. Whole Room / Summary: 더미 통계와 실패 콘텐츠 세계관

데모 기본 사용자는 `현기 / 취준 인간 / 코테, 운동, 답장 자주 실패 /
적당히 킹받게`입니다. 기본 결과는 `SR` 또는 `SSR`로 둡니다. `LEGEND`는
예비 임팩트 카드로만 사용합니다.

## 작업 원칙

- PRD 우선: 새 동작을 만들기 전에 PRD 요구와 연결합니다.
- P0 우선: 데모 사용자 진입, 온보딩 선택, 오늘 상태 선택, 뽑기 연출,
  AI/fallback 카드, 미션 보상, 더미 친구방 피드를 먼저 완성합니다.
- 데모 안정성 우선: 네트워크나 AI 호출이 실패해도 데모 모드, 고정 시드,
  고정 등급, fallback 카드가 동작해야 합니다.
- 작고 읽히는 구현: 에이전트가 로컬에서 읽고 디버깅하기 쉬운 단순한
  Flutter/Dart 구조를 선호합니다.
- 화면으로 증명: 기술 설명보다 사용자가 보는 카드, 연출, 보상, 피드가 먼저
  동작해야 합니다.

## 초기에 과구현하지 말 것

- P0 루프가 완성되기 전에는 Google 로그인, Firebase, Firestore, 실제 친구 초대,
  결제, 랭킹, 사진 인증을 추가하지 않습니다.
- 첫 화면을 랜딩 페이지로 만들지 않습니다. 첫 유효 화면에서 바로 가챠 콘셉트가
  보여야 합니다.
- P0에서는 데모 데이터를 외부 서비스 뒤에 숨기지 않습니다.
- 친구방은 실제 소셜 네트워크가 아닙니다. P0에서는 더미 친구 3~5명과 로컬
  리액션 카운트만으로 공유 가능성을 보여줍니다.
- 전체방 통계는 P0에서 더미 데이터입니다. 실제 통계처럼 설명하거나 구현하지
  않습니다.
- 실패 문구는 죄책감이나 모욕이 아니라 가벼운 농담이어야 합니다.

## AI 카드와 문구 안전 기준

- AI 결과는 JSON 형태의 카드 데이터로 다룹니다.
- 필수 필드는 등급 제목, 설명, 오늘의 미션, 성공 뱃지, 실패 칭호, 실패 메시지,
  공유 문구입니다.
- 미션은 오늘 안에 실제로 할 수 있는 짧은 행동이어야 합니다.
- 우울, 자해, 질병, 외모, 신체, 성별, 인종, 장애, 가족사는 농담 소재로 쓰지
  않습니다.
- AI 응답이 비어 있거나 너무 길거나 파싱에 실패하면 클라이언트에서 기본 문구와
  fallback 카드를 사용합니다.

## 검증 명령

- `make get`: Flutter 의존성을 설치합니다.
- `make format`: Dart 코드를 포맷합니다.
- `make format-check`: 포맷 변경이 필요하면 실패합니다.
- `make analyze`: 정적 분석을 실행합니다.
- `make test`: Flutter 테스트를 실행합니다.
- `make check`: 포맷 확인, 정적 분석, 테스트를 모두 실행합니다.

코드 구현을 넘기기 전에는 기본적으로 `make check`를 실행합니다. 문서만 수정했고
코드 경로가 바뀌지 않았다면 예외로 둘 수 있습니다.

## 문서 지도

- `docs/product/`: 제품 요구사항과 데모 범위
- `docs/architecture/`: 앱 레이어링과 의존 방향
- `docs/exec-plans/`: 실행 계획과 의사결정 기록
- `docs/quality/`: 품질 게이트와 수용 기준
- `docs/reliability/`: 데모 안정성, fallback, 실패 대응 규칙
- `docs/references/`: 하네스 설계에 사용한 참고 자료
