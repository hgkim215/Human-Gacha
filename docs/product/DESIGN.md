# DESIGN.md - 앱인토스 TDS 분석 및 Human-Gacha 적용 기준

작성일: 2026-04-25  
대상: Human-Gacha Flutter 데모, 앱인토스 제출 가능성을 고려한 UI 기준  
주요 근거: 앱인토스 개발자센터 TDS 문서, TDS Mobile 문서, 피그마/앱빌더/UX 가이드

## 1. 결론

앱인토스 TDS는 "예쁜 UI 모음"이 아니라 토스 앱 안에서 실행되는 미니앱이 토스와 충돌하지 않게 만드는 운영 기준이다. 따라서 Human-Gacha의 디자인은 B급 가챠 감성을 살리더라도 화면 골격, 액션, 문구, 접근성은 TDS 관습을 우선해야 한다.

Human-Gacha에 적용할 핵심 방향은 다음과 같다.

- `Navigation -> Top -> Content -> BottomCTA` 흐름을 기본 화면 구조로 둔다.
- 홈, 상태 선택, 결과, 미션, 친구방은 모두 375px 기준의 모바일 화면으로 설계한다.
- 커스텀 카드와 가챠 연출은 제품의 핵심 장면에만 사용하고, 나머지 목록/상태/선택 UI는 TDS 컴포넌트 패턴으로 정리한다.
- 문구는 해요체, 능동형, 긍정형을 기본으로 한다. B급 농담은 가능하지만 사용자를 몰아붙이거나 모욕하는 톤은 금지한다.
- 공식 TDS 패키지는 WebView/React Native 중심이다. 현재 Flutter 구현은 "TDS를 직접 사용하는 것"이 아니라 "TDS 규칙을 참고해 재현하는 프로토타입"으로 봐야 한다.

## 2. TDS가 해결하는 문제

TDS는 토스 제품 전반에서 사용하는 공통 디자인 언어다. 앱인토스에서는 파트너사가 만든 앱이 토스 앱 내부에서 실행되기 때문에, 사용자는 미니앱에서도 토스 수준의 일관성, 예측 가능성, 접근성을 기대한다.

TDS의 실질적 역할은 세 가지다.

- 최소 품질 보장: 커스텀 UI가 흔들려도 Navigation, Top, ListRow, Button 같은 기본 골격은 안정적으로 유지한다.
- 생산성 향상: 화면을 새로 디자인하기보다 검증된 컴포넌트를 조합한다.
- 검수 리스크 감소: 브랜딩, 다크패턴, CTA, 탭바, 해상도 기준을 맞추어 앱인토스 심사에서 문제될 가능성을 줄인다.

## 3. 라이선스와 사용 범위

TDS Mobile UI Kit과 관련 자산은 앱인토스용 애플리케이션 개발, 앱인토스 내 디자인 작업, 앱인토스용 프로토타입 제작 범위에서만 사용하는 것이 안전하다.

주의할 점은 다음과 같다.

- UI Kit, 컴포넌트, 스타일, 브랜드 자산의 권리는 Viva Republica Inc.에 있다.
- 다른 프로젝트, 제품, 서비스에 UI Kit을 재사용하거나 재배포하면 안 된다.
- UI Kit 자체를 복사, 수정, 편집, 재가공해 별도 자산처럼 배포하면 안 된다.
- 앱인토스 제출 목적이 아닌 Flutter 데모에서는 "TDS 공식 자산 사용"과 "TDS 스타일 참고"를 구분해야 한다.

## 4. 화면 구조 원칙

### 기본 구조

대부분의 화면은 아래 구조를 따른다.

```text
Navigation
Top
Content
BottomCTA
```

각 영역의 역할은 명확히 분리한다.

| 영역 | 역할 | Human-Gacha 적용 |
|---|---|---|
| Navigation | 현재 서비스 인지, 닫기/문의/신고/복귀 등 앱인토스 상단 기능 | 모든 화면 최상단에 고정. 데모 풀스크린 연출에서도 영역 침범 금지 |
| Top | 화면 제목, 핵심 설명, 보조 액션, 대표 에셋 | 홈의 오늘의 픽업, 결과 화면 제목, 미션 완료 제목 |
| Content | 리스트, 상태 선택, 결과 카드, 피드 | ListRow/Badge/Button 패턴으로 최대한 단순화 |
| BottomCTA | 다음 행동을 하단에서 고정 제공 | 뽑기, 미션 완료, 공유, 다시 뽑기 같은 주요 액션 |

### 375px 기준

피그마와 앱빌더 가이드는 375px 너비를 기준으로 화면을 설계하라고 안내한다. 앱빌더의 기본 화면은 375 x 812 기준이다. Flutter에서도 다음 기준을 둔다.

- 기준 너비: 375 logical px
- 주요 대응 범위: 360-420 logical px
- 작은 화면에서 CTA, 탭, 긴 문구가 줄바꿈되어도 겹치지 않아야 한다.
- Safe Area가 큰 iPhone 계열에서 Navigation, BottomCTA, Tabbar가 잘리지 않아야 한다.
- 게임식 연출이나 배경 이미지는 레터박스 없이 스케일링하되, 핵심 텍스트는 Safe Area 안에 둔다.

## 5. 파운데이션 해석

### Color

TDS 색상 시스템은 디자이너와 개발자가 같은 색상 이름을 쓰기 위한 장치다. Flutter에서는 공식 `@toss/tds-colors`를 그대로 쓰지 못하므로, 값을 여기저기 하드코딩하지 말고 `AppColors` 같은 단일 토큰 파일로 관리한다.

권장 방향:

- 배경은 흰색/회색 계열을 기본으로 둔다.
- 브랜드 주색은 CTA와 핵심 강조에만 사용한다.
- 등급 색상은 결과 카드와 Badge에 제한적으로 사용한다.
- 빨강/위험 색상은 실패 칭호의 재미 표현이 아니라 실제 위험, 삭제, 오류에만 사용한다.
- 접근성 때문에 브랜드 컬러가 보정될 수 있음을 전제로 대비를 확인한다.

Human-Gacha 색상 역할:

| 토큰 | 용도 |
|---|---|
| `primary` | 뽑기 CTA, 주요 링크, 선택 상태 |
| `background` | 기본 화면 배경 |
| `surface` | 결과 카드, 피드 행, 상태 선택 그룹 |
| `textPrimary` | 제목, 주요 결과 |
| `textSecondary` | 설명, 보조 정보 |
| `borderSubtle` | 섹션 구분, 리스트 구분 |
| `gradeN` - `gradeLegend` | 가챠 등급 표시 전용 |

### Typography

TDS Typography는 구체적인 폰트 크기보다 토큰 계층을 쓰는 것을 권장한다. 값 자체를 외워서 하드코딩하면 큰 글자 접근성 설정에 대응하기 어렵다.

Flutter에서는 다음처럼 의미 기반 스타일을 둔다.

| 스타일 | 용도 |
|---|---|
| `display` | 등급 공개, 결과 카드의 가장 큰 타이틀 |
| `title` | Top 제목, 섹션 제목 |
| `subtitle` | Top 보조 설명, 카드 보조 문구 |
| `body` | 설명문, 피드 본문 |
| `caption` | 확률, 상태, 메타 정보 |
| `button` | CTA와 선택 버튼 |

원칙:

- 화면마다 가장 큰 제목은 하나만 둔다.
- 결과 카드처럼 공유 가능한 장면에서는 큰 타입을 쓰되, 버튼/리스트 내부에는 과한 제목 크기를 쓰지 않는다.
- 시스템 폰트 크기 확대를 막지 않는다.
- 긴 한국어 문구는 2줄까지 자연스럽게 줄바꿈되게 한다.

## 6. 핵심 컴포넌트 분석

| 컴포넌트 | TDS 역할 | Human-Gacha 적용 |
|---|---|---|
| `Navigation` | 서비스 위치와 앱인토스 맥락을 알려주는 최상단 필수 영역 | 모든 화면에 유지. 커스텀 가챠 연출이 침범하지 않게 한다 |
| `Top` | 화면 제목, 대표 에셋, 보조 액션을 배치하는 상단 콘텐츠 | 홈 픽업, 결과 제목, 미션 완료 상태를 표현 |
| `ListRow` | `left`, `contents`, `right`로 나뉜 리스트 항목 | 친구방 피드, 설정, 미션 기록, 보상 목록에 사용 |
| `ListHeader` | 섹션 제목, 설명, 우측 부가 액션 | 친구방, 전체방, 오늘의 상태 섹션 구분 |
| `BottomCTA` | 하단 고정 주요 액션 | 뽑기, 미션 완료, 공유하기, 다시 뽑기 |
| `Button` | 일반 액션 트리거 | 상태 선택, 보조 액션, 데모 모드 제어 |
| `Badge` | 상태를 빠르게 인식하게 하는 강조 요소 | 등급, 확률 버프, 완료/실패 상태 |
| `Border` | 섹션/리스트 경계 표현 | 카드 과다 사용 대신 얇은 구분선으로 밀도 조절 |
| `Asset` | 아이콘, 이미지, Lottie 등 미디어 에셋 표시 | 가챠 캡슐, 등급 아이콘, 미션 배지 이미지 |
| `Paragraph`/`Text` | 본문, 링크, 아이콘, 배지 조합 텍스트 | 결과 설명, 미션 안내, UX 문구 |
| `Tab` | 한 화면 안에서 콘텐츠 전환 | 친구방/전체방, 오늘/지난 기록 전환 |
| `Tabbar` | 하단 주요 기능 이동 | 필요할 때만 사용. 쓰면 반드시 플로팅 형태와 2-5개 제한 준수 |

## 7. 컴포넌트 우선순위

커스텀 UI를 만들기 전에 아래 순서로 대체 가능성을 확인한다.

1. 화면 상단 정보는 `Top`으로 해결한다.
2. 반복 항목은 카드보다 `ListRow`로 먼저 해결한다.
3. 상태 표시는 커스텀 pill보다 `Badge` 패턴을 먼저 쓴다.
4. 주요 액션은 화면 하단 `BottomCTA`로 둔다.
5. 여러 카테고리 전환은 `Tab`으로 해결한다.
6. 진짜 제품 핵심 장면, 즉 가챠 캡슐 연출과 결과 카드만 커스텀한다.

Human-Gacha에서 커스텀이 허용되는 영역:

- 뽑기 캡슐 연출
- 등급 공개 애니메이션
- 결과 카드의 등급/미션 비주얼
- 친구방 공유 카드의 밈적 표현

커스텀이 과하면 안 되는 영역:

- Navigation
- 기본 버튼
- 리스트 행
- 약관/동의/권한 안내
- 오류/로딩/비활성화 상태
- 탭/하단 이동 구조

## 8. UX Writing 기준

앱인토스 UX Writing 가이드는 토스 앱 안의 문구가 일관되게 느껴지도록 해요체, 능동형, 긍정형, 캐주얼한 경어를 강조한다.

Human-Gacha 문구 원칙:

- 기본 문장 끝은 해요체로 쓴다.
- "실패했어요"보다 "다음 뽑기에 쓸 자아 파편을 얻었어요"처럼 긍정형으로 전환한다.
- 버튼은 결과를 예측할 수 있게 쓴다. 예: `뽑기 시작하기`, `미션 완료하기`, `친구방에 공유하기`.
- 다이얼로그 왼쪽 버튼은 `닫기`로 통일한다.
- "하시겠어요", "계신가요" 같은 과도한 경어를 피한다.
- 한자어 명사 나열을 줄인다. 예: `미션 완료 처리`보다 `미션을 완료해요`.

허용되는 B급 톤:

- "오늘은 덜 망한 인간으로 뽑혔어요"
- "도망은 안 간 사람 배지를 얻었어요"
- "카페인 버프로 집중력이 조금 올라갔어요"

피해야 할 톤:

- 사용자를 모욕하거나 무능하게 단정하는 문구
- 건강, 외모, 성별, 가족사, 장애, 질병을 농담 소재로 쓰는 문구
- 실패를 죄책감으로 몰아가는 문구
- CTA 결과가 불명확한 문구

## 9. 다크패턴 방지 기준

앱인토스는 사용자 예상을 벗어난 설계를 치명적인 사용성 오류로 본다. Human-Gacha에서도 다음을 금지한다.

- 앱 진입 직후 광고성 바텀시트나 알림 동의 바텀시트를 띄우기
- 뒤로 가기 시 이탈 방지용 바텀시트를 띄우기
- 닫기/거절/나가기 선택지를 제거하기
- 사용자가 예상하지 못한 순간에 전면 광고나 강제 액션을 노출하기
- CTA 라벨만 보고 다음 화면이나 행동을 예측할 수 없게 만들기

적용 예:

- 첫 화면은 바로 오늘의 픽업과 뽑기 흐름을 보여준다.
- 알림 동의는 미션 완료 후 명확한 혜택 설명과 함께 선택적으로 제안한다.
- 공유하기는 자동 게시처럼 보이게 하지 말고 사용자가 누르는 액션으로 둔다.

## 10. 브랜딩 기준

앱인토스에서는 파트너 서비스가 토스와 명확히 구분되어야 한다. Human-Gacha도 토스처럼 보이되, 토스 자체 서비스처럼 오인되면 안 된다.

브랜딩 체크:

- 앱 이름은 한글 `인간 가챠`를 기본으로 쓴다.
- 브랜드 컬러는 로고/대표 에셋에서 뽑되, 접근성 대비가 낮으면 보정한다.
- 로고는 앱인토스 기준상 600 x 600 정사각형 배경을 가진 형태를 준비해야 한다.
- Navigation과 진입점에서 서비스명이 일관되게 보여야 한다.
- 토스 공식 로고, 토스 브랜드 자산을 Human-Gacha 브랜드처럼 사용하지 않는다.

## 11. 화면별 적용안

### Home

목표: 10초 안에 "오늘 인간을 뽑는 앱"임을 이해시킨다.

구성:

- `Navigation`: 인간 가챠
- `Top`: 오늘의 픽업 인간, 확률 버프, 대표 Asset
- `ListHeader`: 오늘 상태
- 상태 선택 버튼 3-5개
- `BottomCTA`: `인간 가챠 뽑기`

TDS 판단:

- 설명 카드를 여러 개 쌓지 않는다.
- 첫 화면을 랜딩 페이지처럼 만들지 않는다.
- 뽑기 CTA는 항상 명확하게 보이게 한다.

### State Select

목표: 텍스트 입력 없이 1-3개 상태를 고르게 한다.

구성:

- `Top`: 지금 상태를 골라요
- `Button` weak/fill 패턴의 선택 버튼
- 선택된 항목은 `Badge` 또는 selected state로 표시
- `BottomCTA`: `이 상태로 뽑기`

TDS 판단:

- 선택지 문구는 짧고 스캔 가능해야 한다.
- 선택하지 않았을 때 CTA disabled 상태를 제공한다.

### Draw Animation

목표: 등급 공개 전 기대감을 만든다.

구성:

- `Navigation` 영역 보존
- 중앙 Asset/Lottie/커스텀 애니메이션
- 진행 단계 텍스트
- 로딩 시간이 길어지면 fallback 전환

TDS 판단:

- 전체 화면 연출을 하더라도 앱인토스 상단/하단 안전 영역을 침범하지 않는다.
- 로딩 문구는 상태를 알려주고 불안을 줄인다.

### Result Card

목표: 캡처 가능한 핵심 장면을 만든다.

구성:

- `Top`: SR 계획적인 취준 인간 후보
- 커스텀 결과 카드
- `Badge`: 등급, 성공/실패 상태, 오늘 버프
- `Paragraph`: AI 설명과 오늘 미션
- `BottomCTA`: `미션 시작하기`

TDS 판단:

- 결과 카드만 과감하게 커스텀한다.
- 버튼과 보조 액션은 TDS 패턴을 유지한다.
- 긴 설명은 2-3문장 이하로 제한한다.

### Mission Complete / Fail

목표: 성공과 실패 모두 보상으로 이어지게 한다.

구성:

- `Top`: 배지를 얻었어요 / 자아 파편을 얻었어요
- `Asset`: 배지 또는 실패 칭호 이미지
- `ListRow`: 보상 요약
- `BottomCTA`: `친구방에 공유하기`

TDS 판단:

- 실패 화면도 비난하지 않는다.
- 공유는 사용자가 직접 누르는 선택으로 둔다.

### Friend Room / Whole Room

목표: 실제 소셜 구현 전에도 바이럴 가능성을 보여준다.

구성:

- `Tab`: 친구방 / 전체방
- `ListHeader`: 오늘 올라온 결과
- `ListRow`: 친구 이름, 등급, 미션 상태, 리액션
- `Badge`: SR, 성공, 실패 칭호

TDS 판단:

- 피드는 커스텀 카드보다 `ListRow` 기반으로 밀도 있게 만든다.
- 더미 통계는 실제 통계처럼 오해되지 않게 표현한다.

## 12. Flutter 구현 기준

현재 프로젝트가 Flutter이므로 공식 TDS 패키지를 직접 쓰는 방식은 아니다. 대신 다음 구조로 TDS 유사 규칙을 중앙화한다.

권장 파일 방향:

```text
lib/
  design/
    app_colors.dart
    app_typography.dart
    app_spacing.dart
    app_radius.dart
    app_components.dart
```

구현 원칙:

- 색상, 폰트 크기, spacing을 화면 파일 안에 직접 흩뿌리지 않는다.
- `PrimaryButton`, `BottomCta`, `Badge`, `ListRowTile`, `TopSection` 같은 얇은 Flutter 컴포넌트로 반복 패턴을 만든다.
- 결과 카드와 뽑기 애니메이션 외에는 임의의 nested card 레이아웃을 늘리지 않는다.
- 모든 주요 버튼은 loading, disabled, pressed 상태를 가진다.
- Icon-only 버튼은 접근성 label을 가진다.
- 탭, 버튼, 리스트 행은 최소 터치 영역을 확보한다.

## 13. 검수 전 체크리스트

- 모든 주요 화면 최상단에 Navigation 역할이 유지되는가?
- 375px 너비 기준에서 화면이 자연스럽고, 360-420px 범위에서 텍스트가 겹치지 않는가?
- Safe Area와 BottomCTA가 충돌하지 않는가?
- 첫 화면에서 10초 안에 서비스 핵심이 보이는가?
- CTA 라벨만 보고 다음 행동을 예측할 수 있는가?
- 진입 직후 또는 뒤로가기 시 사용자를 막는 바텀시트가 없는가?
- 탭바를 쓴다면 플로팅 형태, 2-5개 제한, 토스 하단 탭과 혼동 방지를 지켰는가?
- 색상과 폰트가 토큰화되어 있고 화면마다 하드코딩되지 않았는가?
- 큰 글자 접근성 설정에서도 주요 정보와 CTA가 보이는가?
- 실패/오류/로딩 상태의 문구가 해요체, 능동형, 긍정형에 가까운가?
- 공식 TDS UI Kit 자산을 앱인토스 외 용도로 재배포하거나 별도 자산화하지 않았는가?
