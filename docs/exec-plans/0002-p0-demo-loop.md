# 0002 P0 데모 루프 구현 계획

상태: planned

## 목표

`docs/product/인간_가챠_PRD_v2_demo_first.md` 기준으로 90초 발표 데모를 안정적으로
재현할 수 있는 P0 제품 루프를 구현한다.

이 계획의 완료 기준은 아래와 같다.

- 앱 실행 후 10초 안에 인간 가챠 콘셉트가 보인다.
- 사용자가 상태 선택부터 뽑기, 결과 카드, 미션 보상, 친구방 피드까지 끊김 없이
  이동할 수 있다.
- AI, 네트워크, 인증이 실패해도 데모 모드와 fallback 카드로 루프가 끝까지
  진행된다.
- SR, SSR, 실패 시나리오를 데모 중 안정적으로 재현할 수 있다.

## 범위

포함:

- Demo User 즉시 진입
- 4문항 이하 온보딩 선택
- 홈의 오늘의 픽업 인간, 확률 버프, 뽑기 CTA
- 오늘 상태 선택 1~3개
- 3단계 이상 뽑기 연출
- AI 카드 생성 인터페이스와 5초 이내 fallback 카드 표시
- 결과 카드, 미션 완료/실패 보상 화면
- 더미 친구방 피드와 전체방 요약
- Demo Mode, 고정 시드, 결과 시나리오 선택
- P0 루프 중심 테스트와 검증 보강

제외:

- Google 로그인
- Firebase Auth, Firestore
- 실제 친구 초대, 실제 리액션 저장
- 실시간 채팅
- 사진 인증, 푸시 알림, 결제, 랭킹

## 주요 결정

- 첫 구현은 `데모 안정성 > 실제 운영성` 우선으로 둔다.
- P0 데이터는 로컬 결정형 데이터로 유지하고 외부 서비스 연결을 전제하지 않는다.
- 앱의 기본 진입은 랜딩 페이지가 아니라 홈 또는 데모 진입 화면으로 둔다.
- AI 카드 생성은 항상 JSON 계약을 통과한 카드 모델로만 UI에 전달한다.
- AI 응답 실패, 타임아웃, 파싱 오류, 과도한 길이는 모두 동일한 fallback 경로로
  흡수한다.
- 친구방/전체방은 P0에서 "실제 소셜"이 아니라 "발표용 무대"로 명시한다.
- `LEGEND`는 구현하더라도 기본 데모 시나리오에 넣지 않고 보조 카드로 둔다.

## 구현 개요

### 1. 앱 셸과 라우팅 정리

- `lib/main.dart`는 앱 부트스트랩만 유지한다.
- `lib/src/app`에 라우팅, 테마, 데모 모드 초기값을 둔다.
- 첫 화면은 홈 중심 플로우로 연결하고, 로그인 화면은 P0 경로에서 제거한다.

### 2. P0 기능 단위 구조 만들기

- `lib/src/features/onboarding`
- `lib/src/features/home`
- `lib/src/features/draw`
- `lib/src/features/result_card`
- `lib/src/features/mission`
- `lib/src/features/social_feed`
- `lib/src/features/demo_mode`
- 공통 모델과 상수는 `lib/src/shared`로 올린다.

### 3. 데모 상태와 시드 모델 설계

- 데모 사용자 기본값: `현기 / 취준 인간 / 코테, 운동, 답장 자주 실패 / 적당히 킹받게`
- 기본 오늘 상태: `잠 부족`, `카페인 필요`, `코테 준비 중`
- 데모 시나리오: `SR 기본`, `SSR 임팩트`, `실패 보상`
- 결과 재현을 위해 등급, 카드 문구 세트, 피드 노출까지 하나의 시드 객체로 묶는다.

### 4. 온보딩과 오늘 상태 선택 구현

- 모든 입력은 선택형 버튼으로만 구성한다.
- 온보딩은 4문항 이하, 30초 내 완료 가능한 밀도로 설계한다.
- 오늘 상태 선택은 1~3개 제한과 즉시 다음 단계 이동을 보장한다.

### 5. 홈 화면 구현

- 첫 화면에서 아래 요소가 동시에 보여야 한다.
  - 오늘의 픽업 인간 2~3개
  - 확률 버프
  - 오늘의 자아 캡슐 또는 메인 카드
  - 인간 가챠 뽑기 버튼
- 설명 텍스트보다 카드/버튼/배지 형태의 시각 정보가 먼저 보이게 한다.

### 6. 뽑기 연출 구현

- 최소 3단계 연출로 기대감을 만든다.
  - 인간성 스캔
  - 등급 실루엣 또는 확률 흔들림
  - 최종 등급 공개
- 데모 모드에서는 연출 길이를 과도하게 늘리지 않고 발표 템포를 유지한다.
- 결과 화면은 캡처 가능한 카드 장면을 기준으로 레이아웃을 잡는다.

### 7. AI 카드 생성과 fallback 체계 구현

- 카드 모델 필수 필드:
  - 등급 제목
  - 설명
  - 오늘의 미션
  - 성공 뱃지
  - 실패 칭호
  - 실패 메시지
  - 공유 문구
- AI 어댑터는 선택 상태와 온보딩 결과를 입력으로 받는다.
- 응답이 5초를 넘기거나 JSON 파싱에 실패하면 fallback 카드로 즉시 전환한다.
- fallback 카드는 SR/SSR/실패 시나리오별로 미리 내장한다.

### 8. 미션 보상 루프 구현

- 결과 카드에서 바로 미션 성공/실패 분기로 이동할 수 있게 한다.
- 성공과 실패 모두 별도 보상 장면을 가진다.
- 성공 시 뱃지, 실패 시 칭호와 자아 파편을 보여준다.
- 어떤 분기에서도 친구방 또는 전체방으로 자연스럽게 이어지게 한다.

### 9. 더미 소셜 무대 구현

- 친구방은 3~5명의 고정 더미 친구와 리액션 카운트를 사용한다.
- 방금 뽑은 내 결과가 자동 공유된 것처럼 피드를 구성한다.
- 전체방은 더미 통계와 실패 콘텐츠 세계관을 보여주는 요약 화면으로 둔다.
- 더미 데이터라는 점은 코드와 문서에서 숨기지 않는다.

### 10. 품질 게이트와 테스트 보강

- 최소 단위 테스트:
  - 데모 시드 선택 로직
  - 카드 fallback 결정 로직
  - 상태 선택 제한
- 위젯 또는 통합 테스트 후보:
  - 홈에서 뽑기 버튼 노출
  - 뽑기 후 결과 카드 진입
  - 미션 성공/실패 후 보상 화면 진입
- 구현 완료 전 `make check`를 통과시킨다.

## 검증

- 수동 검증
  - 앱 실행 후 10초 안에 홈의 핵심 요소가 보이는지 확인
  - 30초 안에 상태 선택과 뽑기 시작이 가능한지 확인
  - SR, SSR, 실패 시나리오가 각각 재현되는지 확인
  - AI 실패를 강제로 만들었을 때 fallback 카드가 5초 이내 노출되는지 확인
  - 미션 성공/실패 모두 친구방 또는 전체방까지 이어지는지 확인
  - 결과 카드, 성공 화면, 친구방 피드가 screenshotable moment인지 확인
- 자동 검증
  - `make format-check`
  - `make analyze`
  - `make test`
  - `make check`
